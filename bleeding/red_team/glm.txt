### 1. The Diamond Graph Memory Corruptor (The ResNet Test)
You proved the memory planner works on a 20-node *sequential* chain. That's easy. Memory planners die on "skip connections" (ResNets).
*   **The Task:** Write `test_diamond_memory.c` that builds this graph:
    `Input -> Conv1 -> ReLU -> Add -> Output`
    `Input ----------------------^`
*   **The Threat:** The memory planner might decide to overwrite `Input`'s buffer to allocate memory for `ReLU`, but `Add` still needs to read `Input`.
*   **The Proof:** Attach the memory pool, execute the graph, and verify the output is mathematically correct. If it segfaults or outputs garbage, your memory planner has a liveness bug.

### 2. The Trained Model Reality Check
Your `audit_pytorch_parity.py` used `np.random.randn` weights. That proves your *kernels* match. It does not prove your *deployment pipeline* works.
*   **The Task:**
    1. Write a 10-line PyTorch script to train `export_pytorch_onnx.py`'s LeNet on CIFAR-10 for **exactly 1 epoch**.
    2. Save the state dict.
    3. Export to ONNX.
    4. Run it through `onnx_to_lancius.py`.
    5. Run it in `run_edge`.
*   **The Proof:** Feed the exact same image into PyTorch and into Lancius. Do they predict the *exact same class*? (Don't worry about logit magnitude, just Argmax class match). Real weights have very different distributions than random weights.

### 3. Kill the MNIST Negative Loss Bug (Once and For All)
This has been hiding in the background since v8p5. You cannot ship V1.0 with a training tutorial that prints `Loss: -21.0407`. It destroys trust instantly.
*   **The Task:** Open `examples/train_mnist.c` and `src/math/lancius_autodiff.c`. Trace the Cross-Entropy (or MSE) implementation.
*   **The Proof:** `./train_mnist` must print a positive, decreasing loss. If it's a signed error in the loss calculation, fix it. If it's just a negative scaling factor in the print statement, fix the print statement.

### 4. Show Me the NaN Injection Output
You added `audit_nan_injection.c` in v9S, but I have never seen it execute. Injecting NaN into a forward pass is easy; injecting it into a *backward pass* is where autodiff engines explode.
*   **The Task:** Ensure `audit_nan_injection.c` runs the full autodiff backward pass over the NaN-injected weights.
*   **The Proof:** Run it. I want to see the terminal output proving it didn't hang, didn't segfault, and correctly propagated the NaN to the gradient without corrupting the arena.

### 5. FFI Error Boundary Validation
You built a beautiful opaque-pointer API (`lancius_stable_api.h`). But what happens when a user makes a mistake?
